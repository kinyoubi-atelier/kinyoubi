## Section 3 - Integrity

### Sitemap

Live sitemap at kinyoubiatelier.com/sitemap.xml lists **22 URLs**. The
`app/sitemap.ts` in the repo also generates 22 URLs. There is **no drift**
between the two.

However, the SEO plan expected 24 URLs. Two URLs are absent from `sitemap.ts`:

- `/work/legal-drafting-methodology` exists as a full page in the repo
  (`app/work/legal-drafting-methodology/page.tsx`) but is not listed in
  `sitemap.ts`. This means Googlebot will not discover it via the sitemap.
- A second URL is unaccounted for against the plan's 24-URL target. This may
  reflect a notes piece that was planned but not yet written, or a section
  index page that was omitted. No page file exists for it in the current repo.

**Action required:** add `/work/legal-drafting-methodology` to `sitemap.ts`.

### robots.txt

The live robots.txt (served by `app/robots.ts`) is **correct**:

- Sitemap URL: `https://kinyoubiatelier.com/sitemap.xml` (right domain)
- General crawl: allowed
- AI training crawlers (GPTBot, CCBot, Google-Extended, ClaudeBot, anthropic-ai,
  ChatGPT-User): disallowed (per copyright policy)

There is a stale static file at `public/robots.txt` that contains the wrong
sitemap URL (`https://kinyoubi.com/sitemap.xml`, truncated domain). Next.js
routes `app/robots.ts` takes precedence over `public/robots.txt` in the app
router, so the wrong file is **not** being served live. But it is a confusing
artefact in the repo and should be deleted.

### JSON-LD

The page code at `app/notes/dpdp-for-saas-builders/page.tsx` correctly
instantiates **two** `<JsonLd>` components:

1. A `TechArticle` schema (built via `buildArticleSchema` in `lib/schema.ts`)
2. A `BreadcrumbList` schema (built via `buildBreadcrumbSchema`)

Both schemas are structurally correct in code: the article schema carries
`headline`, `description`, `datePublished`, `author`, `publisher`, and
`mainEntityOfPage`; the breadcrumb schema carries three list items (Home,
Notes, the article title).

Live verification of the rendered HTML was **not possible** via the automated
fetch tool (which converts HTML to markdown and strips `<script>` tags). The
JSON-LD blocks should be present based on the code, but this should be
confirmed manually using Google's Rich Results Test at
`https://search.google.com/test/rich-results` before the next check-in.

## Section 4 - Recommended actions (prioritised)

**1. Submit the sitemap to Google Search Console immediately.**
The site has a correct sitemap at kinyoubiatelier.com/sitemap.xml. If it has
not been submitted via Search Console (Property: kinyoubiatelier.com), do this
now. Submission requests a crawl; without it, the crawl timeline is
unpredictable for a new domain. This is the single highest-leverage action
available today.

**2. Request indexation for the seven priority pages via URL Inspection.**
In Search Console, use the URL Inspection tool to request indexation
individually for the four notes pages and the three case study pages. Google
processes manual requests faster than organic discovery. Order of priority:
- /notes/dpdp-for-saas-builders (clearest content gap, DPDP query)
- /work/bfsi-mis (companion case study, cross-linked from the DPDP piece)
- /notes/python-fuzzy-record-linkage (second clearest gap, rapidfuzz query)
- /notes/or-tools-vs-fet-comparison
- /work/timetable-engine
- /work/archive-automation
- /notes/postgres-rls-multitenant (most competitive query; deprioritise)

**3. Add /work/legal-drafting-methodology to app/sitemap.ts.**
The page exists and is not discoverable via sitemap. Add it with
`changeFrequency: 'monthly'` and `priority: 0.85` to match the other
case study entries.

**4. Delete public/robots.txt.**
The file has a wrong domain and is overridden by `app/robots.ts`. It is
not harmful live, but it is confusing in the repo and should be removed.

**5. Verify JSON-LD manually before the 60-day check-in.**
Run the DPDP notes page through Google's Rich Results Test. Confirm both
the TechArticle and BreadcrumbList blocks render and validate. If either
fails, check that the `JsonLd` component is not being suppressed by a
client-only render boundary.

**6. Acquire one inbound link before the 60-day check-in.**
For a domain with no index history, the first inbound link from an
authoritative domain (a Hacker News submission, a Dev.to cross-post, a
LinkedIn article with a canonical link) significantly accelerates crawl
discovery. The DPDP piece and the rapidfuzz comparison piece are the two
best candidates for this; they target genuine content gaps and are written
to earn a click from an engineering audience, not a compliance audience.
